# yeast_ros2_utilities_node - Issues

## Bugs / Correctness Problems

### 1. `OdometrySample` to_msg/from_msg drops validity fields
In `src/yeast_msg_helper.cpp:145-170`, the `OdometrySample` conversion functions do not transfer `pose_valid`, `velocity_valid`, or `acceleration_valid` because the `OdometrySample.msg` lacks these fields (schema issue). The `to_msg` function silently drops validity information, and `from_msg` returns an `OdometrySample` with all `*_valid` fields set to their defaults (`false`), even though the data may be valid.

### 2. `SwerveModuleCommand` conversion drops `accel` field
In `src/yeast_msg_helper.cpp:303-327`, `to_msg` and `from_msg` for `SwerveModuleCommand` only handle `speed` and `theta`. The `accel` field from the C++ type is silently dropped on `to_msg` and defaults to 0 on `from_msg`.

### 3. All conversion functions take parameters by value
Every `to_msg` and `from_msg` function takes its parameter by value (e.g., `to_msg(yeast_motion::MotionState yeast_type)` at line 277). For compound types like `MotionState` (which contains two `MotionSample`s, each with `Pose2D`, `Twist2D`, etc.), this causes deep copies on every conversion call. Should be `const&`.

### 4. Hardcoded absolute path to yeastcpp build artifact
`CMakeLists.txt:98`: `${THIRD_PARTY_LIB_FOLDER}/yeastcpp/build/libyeastcpp.a` is a hardcoded absolute path to a build artifact. This breaks if the build directory changes, if building on a different machine, or if using a different build configuration.

### 5. Hardcoded `THIRD_PARTY_LIB_FOLDER` path
`CMakeLists.txt:14`: `set(THIRD_PARTY_LIB_FOLDER "/mnt/working/third_party_libs")` hardcodes an absolute path specific to one machine/workspace layout.

### 6. `include_directories(SYSTEM /usr/local/include/wpimath)` is hardcoded
`CMakeLists.txt:81`: System include path is hardcoded to `/usr/local/include/wpimath`.

## Code Smells

### 7. Generated code is committed but may become stale
`src/yeast_msg_helper.cpp` and `include/yeast_ros2_utilities_node/yeast_msg_helper.hpp` are generated by the Jinja templates but are committed to the repo. If the schemas change and the generator is not re-run, the committed code will be out of sync with the schemas.

### 8. `render_jinja.py` has a leftover comment at the bottom
`template/render_jinja.py:76-77`: Contains a commented-out command-line example that references different arguments than the script actually accepts.

### 9. `package.xml` references `ck_ros2_base_msgs_node` as a dependency
`package.xml:14`: This package depends on `ck_ros2_base_msgs_node` but it is unclear why a utilities node for yeast messages needs this dependency. The CMakeLists.txt also lists it in `ros_find_package_list` (line 28).
